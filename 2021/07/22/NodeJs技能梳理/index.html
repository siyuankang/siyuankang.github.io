事件循环的原理

Node.js 中最基础也是最核心的部分
浏览器的事件循环和nodejs循环原理是不同的
nodejs10+版本之后，虽然在运行结果上与浏览器一致，但二者在原理上一个是基于浏览器，一个是基于libuv库。浏览器的核心是宏任务和微任务，而在NodeJs还有阶段性任务执行阶段。
Node.js 事件循环
事件循环通俗来讲就是一个无限的while循环。现在假设你对这个while循环什么都不了解，你一定会有一下疑问。

  1. 谁来启动这个循环过程，循环条件是什么？
  2. 循环的是什么任务呢？
  3. 循环任务是否存在优先级概念？
  4. 什么进程或者线程来执行这个循环？
  5. 无限循环有没有终点？
    带着这些问题，我们先来看看Nodejs官网提供的事件循环原理图。

NodeJs事件循环原理图

这个过程包括6个阶段，每个阶段代表的含义解释：

  1. timers：本阶段执行已经被setTimeout()和setInterval()调度的回调函数，简单理解就是由这两个函数启动的回调函数。
  2. pending callbacks：本阶段执行某些系统操作（如TCP错误类型）的回调函数。
  3. idle、prepare：仅系统内部使用，你只要知道这2个阶段即可。
  4. poll：检索新的I/O事件，执行与I/O相关问题的回调，其他情况NodeJs将在适当的时候在此阻塞。这也是最复杂的一个阶段，所有的时间循环以及回调处理都在这个阶段执行，接下来会详细分析这个过程。
  5. check：setImmediate()回调函数在这里执行，setImmediate并不是立马执行，而是当事件循环poll中没有新的事件处理时就执行该部分，如下代码：
