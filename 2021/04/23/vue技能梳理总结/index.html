访问登录页时，加载login组件，首页和详情页公共部分提取到了layout组件中，嵌套路由在使用时会把第一层的path和children中的path进行合并，再分别加载对应的layout组件和里面的index组件，detail组件，当path是根路径时，首页可以写成’’，配置children中的path可以是相对路径或者绝对路径，当访问根路径的时候，会先加载layout组件然后再加载index组件，最后合并到一起渲染出来。

编程式导航

我们之前做跳转的时候使用的是router-link超链接，如果我们登录的时候需要点击按钮跳转到首页，就需要使用编程式导航了
实现方法有多种，那它们有什么区别呢？
$router.replace(‘’)不会记住历史，即跳转之后当前页就是浏览器的第一页(从浏览器的左上方前进和后退箭头可以看出)。
$router.push()有2种传参方法，

  1. $router.push(‘/‘)
  2. $router.push({name: ‘Login’,params: { id: 1 } }) 这里的name就是指定义路由规则的时候我们可以给组件定义name
  3. $router.go(-1) 后退1页 也可以使用$router.back()|| $router.go(-2)后退2页

hash模式和history模式的区别

  1. 首页2种模式都是客户端路由的实现方式，路径发生变化后不会向服务器发出请求，使用js监视路径的变化，根据不同的地址渲染不同的内容。
  2. 表现形式的区别
    hash模式：https://baidu.com/#/vedio?id=23232
    history：https://baidu.com/vedio/id/23232

  3. 原理区别
    hash模式是基于锚点，以及onhashchange事件
    history模式是基于HTML5中的historyAPI
    A. history.pushState() 客户端路由，只会改变浏览器地址栏中的地址，并记录到历史记录中，ie10+才支持
    B. history.replaceState() 调用时,路径发生变化，要去请求服务器。

  4. history如何去使用
    history需要服务器的支持
    因为单页页面中，服务端不存在http://www.testurl.com/login这样的地址会返回找不到该页面
    在服务端应该除了静态资源外都返回单页应用的index.html。

history[node服务器配置]

node服务开启后，浏览器输入localhost:5000，就可以打开我们开发的项目首页了。
点击登录和视频，都可以正常访问到对应的组件，视频展示的是404组件。
那么我们点击这些链接时，浏览器发生了什么呢？
我们当前设置的是history模式，history是基于historyAPI中的history.pushState()或者history.replaceState()，点击超链接时，会去调用history的pushState,此方法回去改变地址栏中的地址，并记录到历史记录中去，但是并不会发送请求，以上操作是在客户端中完成的。
当我们F5刷新浏览器时，浏览器就要向服务端发出请求了。请求的地址就是当前浏览器中的url，在node服务器中我们没有处理这个请求地址，所以会报出 Cannot GET /vedio。
我们可以去服务端代码中，开启app.use(history())，我们重启服务，再次刷新浏览器，就可以正常展示出组件内容了。这句代码会去判断当前请求的页面服务器中没有，会把单页应用默认的首页返回给浏览器，浏览器判断当前路由地址，就回去加载对应的组件。

history[ngix服务器配置]

1.下载ngix包，解压至c盘。
2.打开命令行工具，启动start ngix.exe
3.浏览器访问localhost。ngix欢迎页面正常显示说明80端口正常使用。
4.将我们打包好的项目代码复制到html文件夹中，访问localhost，页面正常显示。
5.点击视频 链接后，页面就会显示404。
解决问题：修改ngix配置文件。

重启nginx服务，刷新浏览器，页面正常显示。

pushstate方法仅仅是改变地址栏，并把当前地址记录到浏览器访问历史中，不是跳转到指定路径（即浏览器不会发送请求 ）。
通过监听popstate事件可以监听到浏览器操作 历史的变化，在 popstate的事件处理函数中会记录改变后的地址。调用pushstate或者replacestate方法时，不会触发该事件，点击浏览器的前进或者后退按钮或者调用类似back()时才会触发。
当地址改变之后，浏览器根据当前地址找到对应的组件进行重新渲染。
